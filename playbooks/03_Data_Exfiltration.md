## 1. Purpose

The purpose of this playbook is to provide a structured approach for detecting, investigating, containing, and responding to data exfiltration incidents. It helps SOC analysts identify unauthorized data transfers, minimize data loss, and protect sensitive information.

This playbook follows the NIST SP 800-61 Incident Response Framework and is intended for educational purposes. It can also be adapted to meet an organization's security requirements.

---

## 2. Scope

This playbook applies to data exfiltration incidents involving unauthorized access, transfer, or theft of sensitive information from organizational systems. It is intended for SOC analysts, incident responders, and IT security teams responsible for detecting and responding to data security incidents.

The playbook covers the complete incident response process, including detection, containment, eradication, recovery, and post-incident review. It follows the NIST SP 800-61 Incident Response Framework and can be adapted to different organizational environments.

| Role | Responsibilities |
|------|-------------------|

## 4. Severity Classification

| Severity | Description | Example |
|----------|-------------|---------|
| **Low** | Small amount of non-sensitive data exposed. | Unauthorized download of a few internal files. |
| **Medium** | Sensitive data accessed with limited impact. | Confidential employee records accessed. |
| **High** | Large amount of sensitive data transferred. | Customer database copied outside the network. |
| **Critical** | Major data breach affecting critical business information. | Large-scale theft of confidential organizational data. |

## 5. Detection Logic

Data exfiltration can be detected by monitoring unusual file transfers, abnormal network activity, and unauthorized access to sensitive data. SOC analysts should investigate alerts generated by SIEM tools and review system and network logs.

### Detection Indicators

- Large file transfers outside the network.
- Unusual access to sensitive files.
- Data uploads to unknown websites or cloud services.
- Suspicious outbound network traffic.
- SIEM alerts for abnormal data movement.

### Example Splunk Query

```spl
index=network
```

This query can be used to review network events and investigate suspicious data transfer activity.

## 6. Initial Triage

Initial triage helps confirm the incident, assess its impact, and identify the affected systems before taking response actions.

### Triage Steps

1. Identify the affected system and user account.
2. Review data access and transfer logs.
3. Check the source and destination of the data transfer.
4. Determine the type of data involved.
5. Assess the scope and severity of the incident.

### Information to Collect

| Information | Purpose |
|-------------|---------|
| Username | Identify the affected account |
| Source & Destination | Track data movement |
| File Name | Identify affected data |
| Transfer Time | Build the incident timeline |
| SIEM Alerts | Support the investigation |

---
## 7. Containment Actions

The goal of containment is to stop unauthorized data transfer and prevent further data loss.

### Immediate Containment Actions

1. Isolate the affected system from the network.
2. Disable compromised user accounts.
3. Block suspicious IP addresses or domains.
4. Restrict access to sensitive data.
5. Monitor systems for additional suspicious activity.

### Containment Checklist

| Action | Status |
|--------|--------|
| Affected system isolated | ☐ |
| Compromised account disabled | ☐ |
| Suspicious IP blocked | ☐ |
| Data access restricted | ☐ |
| Monitoring enabled | ☐ |


---
## 8. Eradication Steps

Eradication focuses on removing the cause of the incident and securing the affected environment.

### Eradication Steps

1. Remove unauthorized access and suspicious accounts.
2. Reset compromised credentials.
3. Patch identified security vulnerabilities.
4. Scan systems for malicious activity.
5. Verify that unauthorized data access has stopped.

### Eradication Checklist

| Action | Status |
|--------|--------|
| Unauthorized access removed | ☐ |
| Credentials reset | ☐ |
| Security patches applied | ☐ |
| Systems scanned | ☐ |
| Environment verified | ☐ |


---
## 9. Recovery Checklist

Recovery ensures that affected systems and data are restored securely after the incident.

### Recovery Steps

1. Restore affected systems and data from trusted backups.
2. Verify data integrity and system functionality.
3. Re-enable user access after security checks.
4. Continue monitoring for suspicious activity.
5. Document the recovery process.

### Recovery Checklist

| Action | Status |
|--------|--------|
| Data restored | ☐ |
| Systems verified | ☐ |
| User access restored | ☐ |
| Monitoring enabled | ☐ |
| Recovery documented | ☐ |

## 10. Indicators of Compromise (IOCs)

Indicators of Compromise (IOCs) help identify unauthorized access and suspicious data transfer activities during an investigation.

### Common IOCs

| IOC Type | Description | Example |
|----------|-------------|---------|
| Suspicious File Access | Unauthorized access to sensitive files | Confidential files opened unexpectedly |
| Network Activity | Unusual outbound data transfer | Large upload to an unknown IP |
| User Account | Abnormal account activity | Login from an unusual location |
| SIEM Alerts | Alerts for abnormal data movement | Splunk data transfer alert |
| File Transfer | Unexpected file copying or downloads | Bulk download of sensitive files |

### IOC Handling

- Collect logs and security alerts.
- Identify suspicious users or IP addresses.
- Block confirmed malicious activity.
- Continue monitoring for further data transfers.

## 11. Communication Plan

Clear communication helps ensure a timely and coordinated response during a data exfiltration incident.

### Communication Process

| Team / Role | Responsibility |
|-------------|----------------|
| **SOC Analyst** | Report suspicious activity and update the incident status. |
| **Incident Response Team** | Coordinate containment and recovery activities. |
| **System Administrator** | Secure affected systems and restore services. |
| **IT Manager** | Review the incident and approve response actions. |

### Communication Guidelines

- Notify the incident response team immediately.
- Share only verified information.
- Document all important updates and actions.

---
## 12. Evidence Collection

Evidence collection helps identify the source of the data exfiltration and supports the investigation without altering the original data.

### Evidence to Collect

| Evidence Type | Description |
|---------------|-------------|
| System Logs | Logs related to file access and user activity |
| Network Logs | Outbound traffic and file transfer records |
| SIEM Alerts | Alerts generated by Splunk |
| User Activity | Login history and account actions |
| File Details | Names of affected or transferred files |

### Evidence Handling

- Collect relevant logs and alerts.
- Preserve the original evidence.
- Document all findings for investigation.

---
## 13. Post-Incident Review

The post-incident review helps evaluate the response process and identify improvements for future incidents.

### Review Points

- Identify the root cause.
- Review the response actions.
- Document lessons learned.
- Update security procedures if needed.

---
## 14. Lessons Learned

Lessons learned help improve future incident response and strengthen data protection practices.

### Key Improvements

- Improve monitoring of sensitive data.
- Strengthen access controls.
- Train users on data security.
- Update incident response procedures.
