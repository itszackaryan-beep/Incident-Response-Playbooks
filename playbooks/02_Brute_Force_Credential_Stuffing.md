## 1. Purpose
The purpose of this playbook is to provide a structured approach for detecting, investigating, containing, and responding to brute force and credential stuffing attacks. It helps SOC analysts and IT security teams quickly identify unauthorized login attempts, reduce the risk of account compromise, and protect organizational systems from unauthorized access.
---
## 2. Scope
This playbook applies to brute force and credential stuffing attacks targeting user accounts, authentication services, web applications, VPNs, remote desktop services (RDP), and other systems that require user authentication. It is intended for use by SOC analysts, incident responders, system administrators, and IT security teams responsible for monitoring and responding to authentication-related security incidents.
---
## 3. Roles and Responsibilities

| Role | Responsibilities |
|------|-------------------|
| **SOC Analyst (Tier 1)** | Monitor authentication alerts, investigate failed login attempts, and escalate confirmed incidents. |
| **Incident Response Lead** | Coordinate the response process and oversee containment and recovery activities. |
| **System Administrator** | Disable compromised accounts, reset passwords, and restore affected services. |
| **IT Manager / CISO** | Approve major response actions and coordinate communication with stakeholders. |
---
## 4. Severity Classification

| Severity | Description | Example |
|----------|-------------|---------|
| **Low** | A small number of failed login attempts with no account compromise. | Failed login attempts from a single IP address. |
| **Medium** | Repeated failed login attempts targeting one or more user accounts. | Multiple failed logins on an employee account. |
| **High** | A successful login after multiple failed attempts or multiple accounts targeted. | Compromised user account through brute force. |
| **Critical** | Large-scale credential stuffing or brute force attack affecting multiple systems or privileged accounts. | Multiple administrator accounts compromised across the environment. |

## 5. Detection Logic

Brute force and credential stuffing attacks are detected by monitoring authentication logs, security alerts, and unusual login activity. SOC analysts should identify repeated failed login attempts, abnormal login patterns, and unauthorized access from suspicious locations or IP addresses.

### Detection Indicators

- Multiple failed login attempts within a short period.
- Successful login after repeated failed attempts.
- Login attempts from unknown or suspicious IP addresses.
- Multiple accounts targeted from the same source.
- Authentication alerts generated by SIEM or security tools.

### Windows Event Logs

| Event ID | Description |
|----------|-------------|
| **4625** | Failed login attempt |
| **4624** | Successful login |
| **4648** | Logon using explicit credentials |

### Example Splunk Query

```spl
index=windows (EventCode=4625 OR EventCode=4624)
```

This query helps SOC analysts monitor authentication events and investigate suspicious login activity.

## 6. Initial Triage

Initial triage helps security analysts verify the incident, assess its impact, and collect important information before taking response actions.

### Triage Steps

1. Identify the affected user account and system.
2. Review login history and authentication logs.
3. Check the source IP address and login location.
4. Determine whether the login attempts were successful or failed.
5. Identify if multiple accounts are being targeted.
6. Assign the appropriate incident severity level.

### Information to Collect

| Information | Purpose |
|-------------|---------|
| Username | Identify the affected account |
| Source IP Address | Trace the origin of login attempts |
| Login Time | Build the incident timeline |
| Event ID | Verify authentication activity |
| Authentication Logs | Support the investigation |

---
## 7. Containment Actions

The primary goal of containment is to stop unauthorized access and prevent further account compromise.

### Immediate Containment Actions

1. Disable or lock the compromised account.
2. Block suspicious IP addresses.
3. Reset passwords for affected users.
4. Enable multi-factor authentication (MFA), if available.
5. Continue monitoring authentication activity.

### Containment Checklist

| Action | Status |
|--------|--------|
| Compromised account disabled | ☐ |
| Suspicious IP blocked | ☐ |
| Password reset completed | ☐ |
| MFA enabled | ☐ |
| Monitoring initiated | ☐ |

---
## 8. Eradication Steps

Eradication focuses on removing the cause of the attack and securing affected accounts to prevent future unauthorized access.

### Eradication Steps

1. Remove unauthorized access and terminate suspicious sessions.
2. Reset passwords for compromised accounts.
3. Review user permissions and remove unnecessary privileges.
4. Apply security updates and fix identified vulnerabilities.
5. Verify that no suspicious login activity remains.

### Eradication Checklist

| Action | Status |
|--------|--------|
| Unauthorized access removed | ☐ |
| Passwords reset | ☐ |
| User permissions reviewed | ☐ |
| Security updates applied | ☐ |
| Environment verified | ☐ |


---
## 9. Recovery Checklist

Recovery ensures that affected accounts and authentication services are restored securely after the incident.

### Recovery Steps

1. Verify that compromised accounts are secure.
2. Restore normal user access after security checks.
3. Monitor authentication logs for unusual activity.
4. Confirm that all security controls are working properly.
5. Document the recovery process.

### Recovery Checklist

| Action | Status |
|--------|--------|
| Accounts secured | ☐ |
| User access restored | ☐ |
| Authentication logs monitored | ☐ |
| Security controls verified | ☐ |
| Recovery documented | ☐ |

## 10. Indicators of Compromise (IOCs)

Indicators of Compromise (IOCs) help SOC analysts identify suspicious authentication activity and investigate possible account compromise.

### Common IOCs

| IOC Type | Description | Example |
|----------|-------------|---------|
| Failed Logins | Multiple failed login attempts | Event ID 4625 |
| Successful Login | Login after repeated failures | Event ID 4624 |
| Source IP | Repeated attempts from the same IP | Suspicious external IP |
| User Account | Multiple accounts targeted | Several usernames from one source |
| SIEM Alerts | Authentication-related alerts | Splunk authentication alert |

### IOC Handling

- Collect authentication logs.
- Identify suspicious IP addresses.
- Block confirmed malicious sources.
- Continue monitoring for repeated login attempts.

## 11. Communication Plan

Clear communication helps ensure a quick and coordinated response during a brute force or credential stuffing incident.

### Communication Process

| Team / Role | Responsibility |
|-------------|----------------|
| **SOC Analyst** | Report suspicious login activity and update the incident status. |
| **Incident Response Team** | Coordinate containment and recovery actions. |
| **System Administrator** | Secure affected accounts and restore normal access. |
| **IT Manager** | Review the incident and approve major response actions. |

### Communication Guidelines

- Notify the incident response team immediately.
- Share only verified information.
- Record all important actions and updates.


---
## 12. Evidence Collection

Evidence collection helps security analysts investigate the attack and identify its source without modifying the original data.

### Evidence to Collect

| Evidence Type | Description |
|---------------|-------------|
| Authentication Logs | Login success and failure events |
| Source IP Address | Identify the origin of the attack |
| User Account | Affected or targeted accounts |
| SIEM Alerts | Security alerts generated by Splunk |
| System Logs | Windows Event Logs related to authentication |

### Evidence Handling

- Collect relevant logs and alerts.
- Preserve the original evidence.
- Document all findings for investigation.

---
## 13. Post-Incident Review

The post-incident review helps the security team evaluate the response process and identify areas for improvement.

### Review Points

- Identify the root cause of the attack.
- Review the actions taken during the incident.
- Document lessons learned.
- Update security procedures if required.

---
## 14. Lessons Learned

Lessons learned help improve future incident response and strengthen account security.

### Key Improvements

- Strengthen password policies.
- Enable multi-factor authentication (MFA).
- Improve monitoring of login activity.
- Provide regular security awareness training.
